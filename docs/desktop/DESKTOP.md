# Сайт + приложение для Windows

## Как это устроено

- **Сайт** — обычный веб-интерфейс в браузере (Vite + React).
- **Приложение для ПК** — тот же интерфейс в окне **Electron**; пользователь скачивает установщик **`.exe`** и ставит его как обычную программу.

Страница с объяснением для пользователей: **`/#/download`** (в приложении и на сайте).

Обновление интерфейса портала (glass v2): **[PORTAL-DESIGN-USER.md](../design/PORTAL-DESIGN-USER.md)**.

## Сборка установщика (для разработчика)

В папке проекта:

```bash
npm install
npm run electron:build
```

Скрипт по шагам: подставляет недостающие PNG в `public/` (если их ещё нет), собирает фронт, **собирает сервер** (`server/dist`), обрезает dev-зависимости сервера (`npm prune --omit=dev`), упаковывает основное приложение в **`packaged-app/win-unpacked`**, затем собирает **отдельный неоморфный установщик** (папка **`setup-wizard/`**): это отдельное окно Electron в стиле soft-ui, без классического NSIS. Готовый файл копируется в **`release/Трасса Setup 0.1.0.exe`** (промежуточные файлы — в **`release/setup-dist/`**). После сборки восстанавливается полный `server/node_modules` для разработки.

Пользователь запускает **`Трасса Setup 0.1.0.exe`**, выбирает папку (по умолчанию `%LOCALAPPDATA%\Трасса`), нажимает «Установить» — файлы копируются из встроенного пакета (`payload`).

Значок приложения и установщика: **`electron-assets/icon.png`** (ваш файл копируется сюда при сборке).

Сборка без подписи кода Windows (`signAndEditExecutable: false`), чтобы не требовались права на симлинки в кэше electron-builder. Для публичного распространения позже можно подключить код-подпись.

## Уведомления об обновлении

Используется **не** встроенный NSIS auto-updater (установщик свой), а проверка **JSON по HTTPS**.

Установленное приложение раз в сутки (и при возврате в окно) запрашивает манифест. Если **`version`** в манифесте **новее**, чем в установленной копии, пользователь видит:

1. Системное уведомление Windows «Доступно обновление „Трасса“».
2. Диалог **«Обновить»** / **«Позже»** — по кнопке «Обновить» открывается ссылка на новый **`trassa-setup.exe`**.

### Как выпустить новую версию (кратко)

1. В **`package.json`** увеличьте **`"version"`** (например `0.2.0` → `0.2.1`).
2. В **`release-config.json`** укажите, откуда приложение читает манифест (GitHub Releases или свой сайт):

```json
{
  "publicBaseUrl": "",
  "manifestUrl": "https://github.com/ВАШ_ЛОГИН/ВАШ_РЕПО/releases/latest/download/app-update.json"
}
```

3. Соберите установщик: **`npm run electron:build`** → файл **`release/trassa-setup-<версия>.exe`** и копия в **`public/downloads/trassa-setup.exe`**.
4. Опубликуйте:
   - **GitHub:** `git push origin main` — Actions соберёт релиз, если для этой версии ещё нет тега **`v*`** (см. ниже).
   - **Свой сайт:** выложите **`dist/`**, **`public/downloads/trassa-setup.exe`** и **`public/app-update.json`** (генерируется при сборке).

Перед сборкой скрипт **`scripts/apply-release-config.cjs`** (или **`npm run release:config`**) записывает **`electron-assets/update.json`** с **`manifestUrl`** — без HTTPS-адреса проверка в приложении **выключена**.

Если свой сайт без GitHub:

```json
{ "publicBaseUrl": "https://ваш-сайт.ru" }
```

Манифест: `https://ваш-сайт.ru/app-update.json`, установщик: `/downloads/trassa-setup.exe`.

Подробнее — **`deploy/app-update-manifest.example.json`**.

### Вариант: только GitHub (без своего сайта)

1. Создайте репозиторий на GitHub и залейте код (или подключите локальную папку через **git remote add origin …** и **git push**).
2. В **`release-config.json`** укажите прямую ссылку на манифест последнего релиза (подставьте свой логин и имя репозитория):

```json
{
  "publicBaseUrl": "",
  "manifestUrl": "https://github.com/anastasiaakk/trassa-app/releases/latest/download/app-update.json"
}
```

3. Соберите установщик локально **`npm run electron:build`** и раздайте первый **`.exe`** пользователям.

Дальше новые версии выкладывает **GitHub Actions** (файл **`.github/workflows/release-windows.yml`**).

**Обычный способ (без тегов вручную):** в **`package.json`** увеличьте **`"version"`** (например с `0.1.0` на `0.2.0`), сохраните, сделайте **commit** и **`git push origin main`**. Если для этой версии ещё **нет** релиза на GitHub, Actions сам соберёт установщик, создаст **Release** и тег **`v0.2.0`**. Если вы пушите без смены версии, а релиз для текущей версии уже есть — сборка **пропускается** (чтобы не дублировать релизы).

**Дополнительно:** пуш **тега** `v0.2.0` по-прежнему запускает тот же workflow.

Вкладка **Actions** покажет ход сборки; в **Releases** появятся **`trassa-setup.exe`** и **`app-update.json`**. Ручной запуск workflow (**workflow_dispatch**) собирает установщик в артефакты, **без** создания Release.

Ссылка **`…/releases/latest/download/…`** указывает на файлы **последнего** релиза — у пользователей сработает окно «Обновить», когда версия в манифесте станет больше установленной.

## API в установленном приложении

### Режим по умолчанию для установщика: общий сервер организации

Перед **`npm run electron:build`** укажите в **`release-config.json`**:

```json
"apiUrl": "https://api.ваш-домен.ru"
```

Все копии **`.exe`** ходят на **один** API; локальный сервер на каждом ПК **не запускается**. Подробно: **`../deploy/ORG-SERVER.md`**.

### Режим «свой API на каждом ПК» (без синхронизации между людьми)

Оставьте **`apiUrl`** пустым и собирайте без `--installer` или для отладки. В комплект по-прежнему входит **`resources/trassa-api`**; при запуске **Трасса.exe** поднимается **`http://127.0.0.1:4000`**, если есть Node **22+** в PATH. База — в `%AppData%/…/api-data/app.db`.

## Общие данные портала (синхронизация)

При **`VITE_USE_AUTH_API=true`** календарь, документы ассоциаций, карта, техработы, мессенджер и другие общие настройки хранятся в **SQLite на API** (`portal_kv`), а не только в `localStorage` каждого компьютера.

- Клиент каждые ~25 с подтягивает **`GET /api/portal/state`** (компонент `PortalSyncProvider`).
- Изменения администратора на `/services` уходят на сервер с токеном **`X-Trassa-Admin-Token`**.
- Изменения в кабинетах (документы, календарь, сообщения) — через **`PUT /api/portal/user-kv/...`** (нужен вход пользователя или админ).

**Первый запуск на сервере:** войдите в админку (`ksenia@trassa.local` / `NastiaAdm8` по умолчанию на API) и нажмите **«Синхронизировать данные с сервером»**, чтобы перенести уже накопленные локальные данные.

Все установки должны собираться с **одним и тем же** `apiUrl` в **`release-config.json`**.

## Запуск Electron в разработке

1. Запустите фронт: `npm run dev` (или `Start-Trassa.bat` без API, если не нужен сервер).
2. В другом терминале: `npm run electron:dev`

Electron сам ищет Vite на портах **5173–5176**.

## Вход через API в десктопе

В собранном приложении страницы открываются как `file://`. Cookie к серверу могут не сохраняться, поэтому после входа токен дублируется в ответе API и хранится в **sessionStorage** (заголовок `Authorization`).

Сервер API по умолчанию: **`http://127.0.0.1:4000`**.

Фронт для production собирается с **`VITE_USE_AUTH_API=true`** (файл **`.env.production`** в корне проекта).
